1. PROBLEMAS ENCONTRADOS

- A credencial administrativa legada "alter ego" ainda era tratada como fallback historico em partes do bootstrap e da documentacao.
- O bootstrap do runtime ainda podia sobrescrever componentes explicitamente injetados com paths vindos do ambiente, causando deriva silenciosa entre runtime de teste e runtime efetivo.
- Fila, memoria semantica e objetivos ainda tinham defaults legados fora de data/, criando ambiguidade operacional.
- Stores vivos historicos continuavam rastreados no Git, o que sujava o repositorio durante uso real.
- O registro de dispositivos vivo ainda podia carregar o device id legado fraco "jarvis-dispositivo-local" a partir de stores antigos.
- Havia residuos antigos de validacao nativa e stores historicos fora do caminho oficial.
- Permaneceram diretorios presos em tests/_tmp com "Access denied", fora do fluxo normal, mas ainda presentes no host.

2. O QUE FOI CORRIGIDO

- Autenticacao forte por padrao:
  - runtime/system_config.py agora resolve token, device confiavel e senha admin por ambiente ou bootstrap seguro.
  - Valores fracos legados nao sao mais aceitos como configuracao efetiva.
  - security/access_control.py passou a validar senha por hash PBKDF2, sem depender de senha previsivel embutida.
  - interface/api/app.py deixou de usar fallback operacional silencioso para token/device fracos.
  - interface/native_client/jarvis_client.py e interface/native_app/config.py passaram a consumir o bootstrap/config efetivo.

- Bootstrap coerente:
  - main.py::bootstrap_runtime() foi corrigido para nao sobrescrever GoalManager, fila, auditoria e outros componentes quando eles ja chegam explicitamente configurados.
  - Isso eliminou a regressao que derrubava os testes de integracao da API e dos relatorios.

- Persistencia unificada:
  - executive_planner/queue.py, memory_system/semantic_memory.py e intent_layer/goal_manager.py passaram a usar data/ como store oficial.
  - Foi adicionada migracao automatica de arquivos legados quando o store oficial ainda nao existe.
  - Escrita atomica com arquivo temporario + replace foi aplicada a stores centrais para reduzir corrupcao parcial.

- Separacao entre codigo e estado vivo:
  - Stores operacionais vivos passaram a ficar em data/, logs/ e reports/, todos ignorados no Git quando apropriado.
  - Arquivos rastreados antigos de estado vivo foram removidos do caminho principal de versionamento.
  - O uso normal do runtime nao adicionou novas diferencas ao git status durante o smoke real deste ciclo.

- Registro de dispositivos endurecido:
  - device/device_registry.py agora remove automaticamente o id legado fraco "jarvis-dispositivo-local" ao carregar stores antigos.
  - O mesmo valor passou a ser rejeitado em novas gravacoes.
  - O store vivo atual foi saneado e esse id nao ficou mais confiavel.

- Higiene documental:
  - README.md, docs/referencia/CHANGELOG.md, docs/referencia/DEPLOY_PTBR.md, docs/referencia/API_PTBR.md, docs/referencia/ARCHITECTURE.md, docs/referencia/system_capabilities_index.md, reports/relatorios_txt/reports/relatorios_txt/SYSTEM_MATURITY_REPORT_PTBR.txt, docs/planos/NEXT_STEPS.md, .env.example e relatorios correlatos foram alinhados com o comportamento real.
