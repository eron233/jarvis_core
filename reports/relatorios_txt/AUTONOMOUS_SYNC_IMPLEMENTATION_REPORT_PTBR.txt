3. PREVENCAO DE CONFLITOS

Mecanismos implementados:
- antes de sincronizacao com escrita Git: git pull --rebase
- se git pull --rebase falhar: rebase abortado, push cancelado, lock liberado, evento registrado
- se a area estiver travada por outro dispositivo: o ciclo nao comita nem faz push
- se houver mudancas fora da area reservada: o commit automatico e bloqueado

Observacao importante:
- o lock de area existe e funciona localmente de imediato
- a visibilidade remota desse lock depende de um ciclo de sincronizacao com escrita Git habilitada
- isso foi mantido assim para evitar churn de historico por commits de lock a cada poucos minutos

4. TESTES OBRIGATORIOS ANTES DE COMMIT

Implementado:

  python -m unittest discover -s tests -v

Comportamento:

- o lock e liberado
- o evento e registrado em log e checkpoint

5. MODO SEGURO E MODO AUTONOMO

Modo seguro padrao:
- o motor inicia integrado ao runtime
- faz observacao e diagnostico Git
- NAO faz pull --rebase + commit + push automaticamente sem opt-in explicito

Motivo:
- automacao Git destrutiva em dois PCs ativos pode sobrescrever trabalho valido ou publicar estado inadequado sem revisao
- essa protecao preserva a seguranca estrutural do repositorio atual

Modo autonomo com escrita Git:
- depende de configuracao explicita por ambiente:
  - JARVIS_AUTONOMOUS_SYNC_ENABLED=1
  - JARVIS_AUTONOMOUS_SYNC_ALLOW_GIT_WRITE=1
  - JARVIS_SYNC_DEVICE_NAME=<pc_esposa(voce)|pc_melhor_amigo>
  - JARVIS_SYNC_AREA=<area>

Com isso habilitado, o motor passa a:
- git fetch
- git pull --rebase
- rodar testes
- reservar/liberar lock
- git add
- git commit
- git push
- registrar checkpoint

- a arvore local ja contem mudancas humanas e historico ativo
- liberar escrita autonoma imediata sem opt-in violaria a meta de endurecimento estrutural
