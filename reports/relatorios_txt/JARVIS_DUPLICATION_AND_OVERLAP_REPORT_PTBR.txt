==================================================
1. DUPLICACOES REAIS ENCONTRADAS
==================================================

1.1 Relatorio geral da API exposto duas vezes com a mesma construcao
Classificacao: DUPLICADO REAL
Arquivos:
- interface/api/app.py
- runtime/internal_agent_runtime.py
Evidencia:
- /api/relatorio chama runtime.build_system_report(...)
- /api/relatorio/sistema chama runtime.build_system_report(...)
- ambos ainda anexam ambiente e retornam essencialmente o mesmo payload
Diagnostico:
- sao duas rotas para a mesma responsabilidade
- isso nao e "flexibilidade"; e alias funcional sem necessidade forte
Risco:
- documentacao, clientes e testes podem divergir sobre qual rota e a oficial

1.2 Multiplicacao de launchers para o app nativo
Classificacao: DUPLICADO REAL
Arquivos:
- jarvis.cmd
- jarvis_native.pyw
- jarvis_native.vbs
Evidencia:
- jarvis.cmd no modo native-app chama jarvis_native.pyw
- jarvis_native.vbs tambem chama jarvis_native.pyw
- os dois entregam a mesma acao final para o usuario: abrir o app nativo
Diagnostico:
- ha pelo menos tres pontos de entrada para a mesma UI nativa: cmd, pyw e vbs
- pyw e o entrypoint tecnico; cmd e vbs viraram wrappers paralelos
Risco:
- suporte e troubleshooting ficam confusos
- um launcher pode ser corrigido e outro continuar obsoleto

1.3 Fluxo de bootstrap do relatorio geral duplicado entre API e runtime
Classificacao: DUPLICADO REAL
Arquivos:
- interface/api/app.py
- runtime/internal_agent_runtime.py
Evidencia:
- a API monta relatorios de sistema chamando metodos dedicados do runtime
- mas ainda carrega aliases de apresentacao praticamente identicos para a mesma informacao
Diagnostico:
- nao e um bug grave, mas e duplicacao de contrato externo
- o usuario externo ganha dois nomes para a mesma coisa

2. SOBREPOSICOES FUNCIONAIS ENCONTRADAS
==================================================

2.1 main.py e runtime/server.py dividem a responsabilidade de "subir o Jarvis vivo"
Classificacao: SOBREPOSICAO FUNCIONAL
Arquivos:
- main.py
- runtime/server.py
Evidencia:
- main.py contem bootstrap_runtime, JarvisSystemLoop e o loop continuo autonomo
- runtime/server.py importa bootstrap_runtime e JarvisSystemLoop de main.py
- runtime/server.py tambem pode iniciar o loop continuo em background via RuntimeLoopWorker
Diagnostico:
- main.py e o entrypoint do loop puro
- runtime/server.py e o entrypoint do servidor HTTP que tambem pode hospedar o loop
- os dois disputam a mesma responsabilidade de "processo vivo principal", ainda que em formatos diferentes
Risco:
- um operador pode subir o loop puro e achar que subiu o sistema completo
- outro pode subir o servidor com loop e criar a falsa impressao de que main.py nao e mais necessario

2.2 jarvis.cmd possui dois caminhos para API/servidor que nao passam pelo mesmo bootstrap
Classificacao: SOBREPOSICAO FUNCIONAL
Arquivos:
- jarvis.cmd
- runtime/server.py
- interface/api/app.py
Evidencia:
- modo server executa runtime/server.py
- modo api-direct executa uvicorn interface.api.app:app diretamente
- create_app consegue bootstrapar o runtime por conta propria em _ensure_runtime_initialized
Diagnostico:
- o modo server usa JarvisServerContext, logging, startup report e configuracao mais completa
- o modo api-direct bypassa esse contexto e vai direto para create_app
- isso cria dois caminhos para a mesma API com niveis diferentes de orquestracao
Risco:
- comportamento operacional diferente para a mesma porta e a mesma API
- identidade de runtime, loop em background, logs e startup report podem divergir

2.3 jarvis_run.cmd sobrepoe o papel de jarvis.cmd server
Classificacao: SOBREPOSICAO FUNCIONAL
Arquivos:
- jarvis.cmd
- jarvis_run.cmd
Evidencia:
- jarvis_run.cmd prepara variaveis, abre navegador e no fim chama jarvis.cmd server
Diagnostico:
- ele nao e um novo servidor; e um launcher amigavel por cima do launcher tecnico
- ainda assim, no dia a dia ele compete como "forma principal de iniciar o Jarvis"
Risco:
- duas instrucoes oficiais para a mesma operacao local

2.4 Tres clientes reais para o mesmo backend

- multiplos logs e relatorios internos para o mesmo dominio de saude estrutural

RESPOSTAS DIRETAS AS PERGUNTAS OBRIGATORIAS

Existe mais de uma forma de iniciar a mesma parte do Jarvis?
- Sim.
- Servidor/API: runtime/server.py, jarvis.cmd server, jarvis_run.cmd, service/jarvis_windows_service.py, interface/native_app/runtime_bootstrap.py e ainda jarvis.cmd api-direct.
- App nativo: jarvis.cmd native-app, jarvis_native.pyw e jarvis_native.vbs.

Existe mais de um store "verdadeiro" para a mesma informacao?
- Para fila, memoria semantica, objetivos, auditoria e device registry vivos: hoje nao. A fonte viva oficial esta em data/.
- Mas ainda ha sombra de legado no codigo e muitos clones historicos/teste em data/, reports/ e tests/.

Existe mais de um modulo tentando coordenar dispositivos?
- Para desenvolvimento remoto: nao. O coordenador real e autonomous_sync_engine com TASK_LOCKS.json.
- Para acesso de clientes: ha device_registry, mas ele trata confianca de dispositivo, nao coordenacao de trabalho entre PCs.

Existe mais de um cerebro/avatar ativo ou parcialmente ativo?
- Sim.
- Um cerebro web em interface/brain_avatar para o dashboard.
- Um cerebro nativo proprio em interface/native_app/brain_widget.py + brain_scene.html/js.

Existe mais de um caminho de autenticacao relevante?
- Sim.
- Gate HTTP por token/device/nonce.
- Sessao de painel por cookie derivado.
- Privilegio admin por AccessControl dentro do comando.

Existe mais de um fluxo de cliente/app?
- Sim.
- Dashboard web.
- Cliente CLI leve.
- App nativo Qt.

Existe mais de um mecanismo de otimizacao/auto-organizacao competindo?
- Sim.
- SelfOptimizationCore.
- SelfImprovementAdvisor.
- RemediationEngine dentro de self_defense.
- Em paralelo, varios orgaos vitais tambem sinalizam problemas estruturais correlatos.

==================================================
CONCLUSAO BRUTAL
==================================================

O Jarvis atual nao esta "duplicado por inteiro", mas esta claramente dividido contra si mesmo em quatro frentes:

1. entrypoints e launchers
2. contratos externos da API
3. renderizacao do cerebro/avatar
4. auto-diagnostico/auto-melhoria interna

Os stores vivos principais ja estao muito mais unificados do que antes. O maior problema de duplicacao hoje nao esta mais em fila/memoria/objetivos; esta em como o sistema sobe, como ele se apresenta por clientes diferentes e como varias camadas internas vigiam ou "melhoram" o mesmo organismo por caminhos paralelos.

Em resumo:
- persistencia viva principal: relativamente unificada
- inicializacao e launchers: ambigua
- clientes: multiplos e sobrepostos
- cerebro visual: dividido em duas stacks
- diagnostico interno: fragmentado
